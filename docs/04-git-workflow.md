# 04 — Git Workflow

Guild enforces a strict git workflow across all managed projects. Every quest has a branch. Every chain ends in a PR. Merges to main always require human approval.

---

## Branch Architecture

Configured automatically on `guild project add`.

| Branch | Rule |
|---|---|
| `main` | No direct push. Human approval required for all merges. |
| `development` | No direct push. Hero reviewer approval required. Tests must pass. |
| `feature/GLD-*` | Heroes push freely. Cannot self-merge. |
| `fix/GLD-*` | Heroes push freely. Cannot self-merge. |
| `chore/GLD-*` | Heroes push freely. Cannot self-merge. |

---

## Full Git Lifecycle

### Step 1 — Quest Created
Guild Master generates a branch name at quest creation. No quest exists without a branch assignment.

```
Quest: Implement Telegram Adapter  →  feature/GLD-042-telegram-adapter
Quest: Fix memory leak in runtime  →  fix/GLD-043-wasm-memory-leak
Quest: Update README docs          →  chore/GLD-044-update-readme
```

### Step 2 — Hero Starts
Hero checks out new branch from `development` — never from `main`. Hero's `CLAUDE.md` is injected with quest context and branch name. Hero commits incrementally throughout — not one large commit at the end.

### Step 3 — Implementation Complete
Hero pushes branch, writes completion report to outbox. Guild Master detects completion, auto-creates a **test quest** assigned to a different hero. Implementor moves to resting status.

### Step 4 — Test Quest
Hero Tester works on the same branch.

- **Tests pass** → Guild Master auto-creates PR from feature branch to `development`
- **Tests fail** → Guild Master creates a fix quest, re-assigns to original implementor. Cycle repeats.

### Step 5 — PR Created
PR body is auto-generated from quest description, test results, and changed files. Guild Master immediately creates a **review quest** assigned to a hero not previously involved in this chain.

### Step 6 — Review Outcome

**Approved:**
Guild Master merges PR to `development`. Branch deleted. All chain heroes receive XP. Developer notified via Telegram. Shared memory updated with learnings.

**Changes Requested:**
Guild Master creates an address-review-comments quest, re-assigns to implementor with review comments attached. Chain continues from Step 2.

### Step 7 — Development to Main

> **Only step requiring human approval.**

Guild Master monitors `development` and sends a Telegram message when ready to merge to `main`. Developer replies with approval. Guild Master creates and merges the PR. This is the only irreversible action in the standard workflow.

## Commit Convention

Heroes follow this commit message format — enforced via CLAUDE.md instruction:

```
[GLD-{id}] {short description} — {hero_name}

# Examples
[GLD-042] Add TelegramAdapter struct — StormForge
[GLD-042] Implement send_message method — StormForge
[GLD-042] WIP — paused by developer — StormForge
```

Guild Master uses commit messages as lightweight progress indicators — no need for heroes to write status reports mid-quest.

## Git Provider Integration

Guild supports GitHub and GitLab for PR management. Configured at project registration.

```bash
# GitHub
guild project add --provider github --path ./my-repo

# GitLab
guild project add --provider gitlab --path ./my-repo

# No provider (local git only — no PR automation)
guild project add --provider none --path ./my-repo
```

Provider credentials stored in guild secrets:

```bash
guild secret add GITHUB_TOKEN ghp_xxxx
guild secret add GITLAB_TOKEN glpat_xxxx
```

When provider is `none`, Guild Master still manages branches locally but cannot auto-create PRs. Developer gets a Telegram notification with the merge command to run manually.
